How
staticrypt
Works.

GitHub Pages × クライアント側AES復号

デモサイトのパスワード保護で使っている staticrypt が内部で何をしているか。 なぜサーバーが要らないのか、なぜパスワードの長さがそのまま暗号強度になるのかを1ページにまとめた。

Updated · 2026-05-10 / VAIZO V/ENTER WEB
結論 · TL;DR

パスワード認証も復号も、すべてブラウザのJavaScriptで完結している。

  • サーバー側認証は無し（GitHub Pagesは静的配信のみ）
  • 暗号化済HTMLは誰でもダウンロード可能（URLを知っていれば）
  • 復号は100%ブラウザJSでAES-256（CryptoJS / Web Crypto API）
  • レート制限不可能 → 攻撃者は手元でJSを動かして総当たりできる
  • つまりパスワードの長さ＝暗号強度になる
01 / Flow

全体フロー ビルドからアクセスまで

ビルド時にHTML本文を暗号化して保存し、ブラウザでJSが復号する2フェーズ構成。 通常のWebサーバーでは「サーバーがPW検証→正しければHTMLを返す」だが、 staticryptは「あらかじめ暗号化したHTMLを誰にでも返す→ブラウザが復号する」というモデル。

[ Build Time · ビルド時（GitHub Actions上）] 元のHTML + パスワード（12文字以上） | v AES-256 暗号化（PBKDF2 で鍵導出） | v 出力されるHTML（1ファイル）の中身： ├─ 暗号化された元HTML（Base64） ├─ staticrypt の復号用JS（数KB） ├─ パスワード入力フォーム └─ HMAC（PW正誤の検証用ハッシュ） | v GitHub Pages の Public repo に push [ Distribution · 配信（GitHub Pages）] https://enter-web.github.io/{案件}-demo/ ← 誰でもアクセス可 | v 暗号化済HTML（≒ ランダムなBase64文字列）が返る | v 攻撃者がここでDLしても、中身は意味不明（暗号化済） [ Access Time · アクセス時（クライアントのブラウザ）] 1. ブラウザが上記HTMLをロード 2. パスワード入力フォームが表示される 3. ユーザーがPWを入力 | v 4. JSが PW から PBKDF2 で鍵を再生成 5. JSが その鍵で暗号文をAES復号 6. HMAC で検証 | ┌─ OK → 復号できた元HTMLを画面に描画 → 閲覧開始 └─ NG → "Bad password" を表示
02 / Key Points

重要ポイント 勘違いしやすいところ

項目実際の挙動
サーバー側認証無し（GitHub Pagesは静的配信のみ）
暗号化済バンドルURLを知っていれば誰でもダウンロード可能（ファイル単独で完結）
復号処理100% ブラウザのJavaScript（CryptoJS or Web Crypto API）
復号アルゴリズムAES-256-CBC（PBKDF2で鍵導出、HMACで検証）
パスワード入力リトライ制限無し（クライアント側なので技術的に不可能）
サーバーログ無し（誰がいつ閲覧したか記録できない）
オフライン総当たり攻撃可能（攻撃者がHTMLを保存して手元のスクリプトで試行）
暗号強度事実上、PWのエントロピーがそのまま暗号強度
復号後のHTMLDOMに直接挿入（メモリ上のみ・ファイルには残らない）
localStorage 保存オプション。「次回以降PW入力をスキップ」したいときのみ有効化
見落としやすい注意 クライアントが F12でDevToolsを開けば 暗号化済バンドルの構造を見られます。中身は復号できなくても、「このサイトは staticrypt で守られている」「使われているライブラリのバージョン」などは丸見え。あくまで「コンテンツの秘匿」が目的で「存在の秘匿」ではないことに注意。
03 / Strength

なぜ12文字以上が必須か PW長 = 暗号強度

オフライン総当たり攻撃を前提とすると、PWの組み合わせ数がそのまま破られるまでの時間になる。 一般的な攻撃マシン（GPU 1枚、毎秒10億回試行）を想定した試算。

VAIZO 基準 12文字 = 約70年の総当たり耐性 をクライアント確認用デモの最低基準としている。 案件期間（数週間〜2ヶ月）に対して十分な強度差。 16文字以上は「機密情報を含むデモ」「VIP案件」で推奨。
参考：他方式との比較 Cloudflare Access や Vercel Password Protection はサーバー側認証のため、入力リトライにレート制限がかかる（5回失敗で1時間ロック等）。 つまり「短いPWでも一定の安全性」が得られる。 一方 staticrypt はレート制限不可能なので、PW長さで殴り勝つしかない。これがstaticrypt運用の最大の制約。
